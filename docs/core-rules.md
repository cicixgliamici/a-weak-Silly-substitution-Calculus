# Core Rules — A Weak Silly Substitution Calculus

This document collects a small set of informal core rules for understanding the calculus presented in the repository.

The aim is not to claim a complete metatheory, but to make the repository immediately more concrete and readable.

---

## 1. Minimal syntax

We consider terms generated by the following grammar:

```text
t ::= x
    | λx.t
    | t t
    | t[x := u]
````

where:

* `x, y, z, ...` are variables
* `λx.t` is abstraction
* `t1 t2` is application
* `t[x := u]` is an explicit substitution object

The main idea is that substitution is represented inside the syntax of terms rather than being only a meta-level operation.

---

## 2. Main intuition

In ordinary presentations of the lambda calculus, beta-reduction is often written as if substitution happened immediately and externally.

Here, the first reduction step can instead expose substitution explicitly:

```text
(λx.t) u  →  t[x := u]
```

This makes substitution visible inside the calculus and allows us to study how it propagates through term structure.

---

## 3. Informal propagation rules

A small family of rules that captures the basic intuition is the following.

### Variable hit

```text
x[x := u] → u
```

If the variable being substituted matches the target variable, the substitution is resolved.

### Variable miss

```text
y[x := u] → y      if y ≠ x
```

If the variable does not match, the substitution has no effect.

### Application propagation

```text
(t1 t2)[x := u] → (t1[x := u]) (t2[x := u])
```

A substitution inside an application can be propagated to both subterms.

---

## 4. Binder caution

The abstraction case is where things become more delicate.

A naive rule such as

```text
(λy.t)[x := u] → λy.(t[x := u])
```

is only safe when variable capture is not introduced.

This means that any more precise account of the calculus should eventually clarify:

* free variables
* bound variables
* freshness conditions
* or the specific mechanism used to avoid capture

For the purposes of a minimal presentation, it is enough to note that the abstraction case requires care.

---

## 5. Why these rules matter

Even this tiny set of rules already shows why explicit substitution calculi are interesting.

They help connect:

* reduction semantics
* rewriting systems
* implementation-oriented views of evaluation
* and the structure of substitution itself as a computational object
